# PRAGMATA 双人合作助手

## 这是什么

《PRAGMATA》是卡普空发布的一款结合了第三人称射击和走格子小游戏的游戏。在走格子部分，你需要用四个方向键来完成任务。这个工具可以把这些操作分给两个人——一个人用一把手柄负责射击，另一个人用二把手柄负责走格子。游戏只看到一把虚拟手柄，而实际上两把物理手柄的输入被合并后发送给了它。

## 依赖安装

你需要在电脑上安装下面这三个东西。

首先是 **ViGEmBus 驱动**。我们的程序需要创建一个虚拟的 Xbox 360 手柄给游戏使用，这个驱动就是干这个的。你有两种方式来安装它：一种是下载 ViGEmBus 的安装包，运行安装程序即可；另一种是用包管理器 `winget`，在命令行中执行 `winget install Nefarius.ViGEmBus`。一定要在运行我们的程序之前完成安装。

其次是 **HidHide 驱动**。这个驱动用来把物理手柄藏起来，不让游戏直接看到它们。同样有两种安装方式：去它的发布页面下载安装包，或者在命令行中执行 `winget install Nefarius.HidHide`。安装完成后需要重启电脑。

最后是 **.NET 8.0 桌面运行时**。我们的程序是用 C# 编写的，需要这个运行时才能跑。去微软的官网下载 .NET 8.0 Desktop Runtime（注意是桌面版，不是 SDK），安装即可。

## 使用说明

第一步，确认你已经用管理员权限打开了程序。在 PragmataCoop.exe 上右键，选择"以管理员身份运行"。HidHide 的配置需要管理员权限，所以这一步不能省略。

第二步，把两个手柄都插上电脑。程序启动后会自动开始检测手柄。你会看到界面上手柄一的区域显示"请转动爸爸手柄的左摇杆"，手柄二的区域显示"请转动女儿手柄的右摇杆"。按照提示操作：转动第一个手柄的左摇杆来标识手柄一，再转动另一个手柄的右摇杆来标识手柄二。检测成功后两个区域都会显示绿色的"已识别"标志。

第三步，选择合适的映射方案。界面上有一个下拉框，提供三种方案。选择"右摇杆"时，手柄二用右摇杆的四个方向映射到虚拟手柄的 Y、A、X、B 键，推荐使用。选择"按键"时，手柄二直接用 A、B、X、Y 四个按键。选择"混合"时，摇杆和按键可以同时使用。

第四步，点击"启动"按钮。程序会创建一把名叫 "PragmataVirtualController" 的虚拟 Xbox 360 手柄，然后通过 HidHide 把两个物理手柄藏起来。你最好在点击启动后把两个手柄重新插拔一下，确保 HidHide 的屏蔽立即生效。

第五步，启动游戏。游戏只会看到那一把虚拟手柄。手柄一控制射击时几乎所有的操作都通过虚拟手柄传给游戏。手柄二的 LT 键是切换开关：按住 LT 的时候，手柄二的右摇杆或 ABXY 键会接管走格子操作；松开 LT 的时候，手柄一恢复完全控制。

当合作模式激活（手柄二按住 LT）时，手柄一的左摇杆和右摇杆、扳机键、肩键、十字键、菜单键仍然正常工作，只有 ABXY 四个键被屏蔽。这两种操作互不干扰。

如果手柄二按住 LT 但走格子操作没反应，试着更用力地按 LT。触发阈值是大约百分之二十五的行程。

需要停止的时候，点"停止"按钮或关闭窗口。程序会自动销毁虚拟手柄并解除 HidHide 的屏蔽。

## 大致原理

这个程序本质上是一个手柄输入的转发器。它同时监听两个物理手柄的输入，按照一套规则把它们合并成一个逻辑手柄的状态，然后把合并后的状态写入一把虚拟手柄。游戏启动的时候只看到这把虚拟手柄，所以它以为只有一个人在玩。

物理手柄的读取用的是 XInput 这个 Windows 自带的手柄 API。程序用 C# 直接调用系统底层的 XInput DLL，把原始的手柄数据读到内存里，然后手动解析。这里有踩过一个坑：直接把 C# 的结构体传过去有时候内存对齐会出问题，所以我们改成了读原始字节再手动用 BitConverter 解析，这样完全不依赖 P/Invoke 的结构体封送。

虚拟手柄的创建用了 ViGEmBus。它是一个内核级的虚拟设备驱动，能够创建和真实设备一模一样的虚拟 Xbox 360 手柄。游戏无法区分它是真的还是假的。我们的程序通过 ViGEm 的 C# 客户端库向这个虚拟手柄写入摇杆位置、扳机力度、按键状态等数据。

两个手柄的输入合并逻辑很简单。当手柄二的 LT 没有被按下时，所有输入都从手柄一来，手柄二除了 LT 用来检测激活之外全被忽略。当手柄二的 LT 被按下进入合作模式时，手柄一的 ABXY 四个键被屏蔽，手柄二根据选择的映射方案接管这四个键。手柄一的摇杆、扳机、肩键等其他输入继续正常通过。

最棘手的问题是怎么让游戏看不到物理手柄。游戏通常会枚举所有连接的手柄，物理手柄和虚拟手柄都会被发现。如果游戏同时能读到物理手柄，那 ABXY 还是能被直接发到游戏里，我们的屏蔽就没有意义了。为了解决这个问题，我们引入了一个内核级的 HID 过滤驱动 HidHide。

HidHide 能够在 Windows 的内核层拦截应用对 HID 设备的访问请求。我们通过它的 API 扫描系统上所有的 HID 和 USB 设备接口，把除了虚拟手柄和键盘鼠标之外的所有设备都加入黑名单。然后我们把自己的程序加入白名单，这样只有我们的程序能看到物理手柄的原始数据。对游戏来说，物理手柄从此"不存在"了。

这里面有一个很微妙的细节：HidHide 必须屏蔽 HID 和 USB 两个层级的设备实例。如果只屏蔽 HID 接口，有些通过 DirectInput 或 USB 驱动直接访问的游戏仍然能看到手柄。所以我们的程序用 Nefarius.Utilities.DeviceManagement 库枚举了 HID 设备和 USB 设备两个层级的接口，把它们一起加入黑名单。

还有一个坑：ViGEmBus 创建的虚拟手柄实例 ID 早先是含有 ROOT 前缀的，后来我们给虚拟手柄自定义了 VID 和 PID，所以只需要简单排除 ROOT 开头的设备实例就可以区分虚拟手柄了。但我们在调试过程中发现，北通宙斯 T6 这类仿 Xbox 的手柄，它的 USB 实例 ID 恰好包含了一个和旧版 ViGEm 序列号一模一样的字符串 DC5B7401。我们之前误把它当成了虚拟手柄给跳过了，导致了手柄二没有被屏蔽。去掉这个错误过滤之后一切正常。

虚拟手柄的自定义 VID/PID 还有一个额外的好处：在 HidHide 配置工具里能一眼认出来，方便手动排查问题。

整个程序的 UI 用 WPF 搭建，ViewModel 通过数据绑定驱动界面更新。检测手柄的状态机有两个阶段：先用左摇杆找手柄一，再用右摇杆找手柄二。两个都找到后用户点击启动，程序创建虚拟手柄，配置 HidHide，然后进入映射循环。映射循环每十六毫秒轮询一次两个手柄的 XInput 状态，计算出合并后的虚拟手柄状态，写入 ViGEm 驱动。

停止的时候顺序相反：先停映射循环，再销毁虚拟手柄，最后关闭 HidHide 屏蔽。这样下次启动可以重新检测和配置。

关于 REFramework 的探索也值得记录一下。在 HidHide 还没有完全弄好之前，我们尝试过用 REFramework 这个 RE 引擎的 mod 框架来从游戏内部分层屏蔽手柄。思路是 Hook 游戏内部的 via.hid.GamePad 类的方法，让游戏只能读到虚拟手柄的数据。但实际测试发现，Pragmata 并不通过 via.hid.GamePad 的 get_Device 或 get_MergedDevice 这些方法来读取手柄输入，Hook 从未被触发。我们转而尝试了 REFramework 的 C# 插件机制和更新框架回调，但 MergedGamePadDevice 类型里没有任何字段可供修改。游戏的手柄读取是 C++ 层直接从 XInput API 拿数据，完全不经过可以被 C# 或 Lua 拦截的方法。这条路走不通，但也帮助我们确认了只能从系统内核级解决屏蔽问题的方向。
